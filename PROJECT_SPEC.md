## 三、回测引擎规则

### 3.1 匹配算法：6维欧氏距离

- engine.js `findNearest6D()` 用 **6 个资产的实际权重** 在 `APP_DATA.trendData[].alloc` 中找最近邻
- **不再使用** 4 维压缩（a/u/g/c），因为会丢失沪深300 vs 中证500、标普500 vs 纳指100 的区分信息
- 距离阈值 `MATCH_THRESHOLD = 0.15`，超过则走 `estimateFromScratch`

### 3.2 精确匹配规则

- 三档方案已注入 trendData（194条中的最后3条），distance=0 精确匹配
- 当用户在滑块中精确设置三档方案的配置时，返回的是 trendData 中预计算的结果
- 其他配置走 `estimateFromScratch`（6资产独立月度收益率加权计算）

### 3.3 falsy 值陷阱

```javascript
// ❌ 错误：c=0 时 0 || 0.20 = 0.20
const dc = params.c - (item.c || 0.20);

// ✅ 正确
const dc = params.c - (item.c != null ? item.c : 0.20);
```

### 3.4 滑块总和锁定规则

- 滑块总和始终 = 100%
- 调整某一项时，其他项按比例自动缩放以保持总和 100%
- **这意味着减现金 ≠ 放活期**，减现金会导致其他资产等比例增加
- 这是当前引擎的设计限制，详见"已知限制"章节

### 3.5 默认配置

```javascript
DEFAULT_CONFIG = {
  '沪深300': 15, '中证500': 5, '标普500': 15,
  '纳斯达克100': 20, '黄金': 20, '现金·货币基金': 25
};
```

## 六、已知限制 & 待解决

### 6.1 算法统一为买入持有（已解决 ✅）

**原因**：恒市值法回测（trendData）和买入持有加权（estimateFromScratch）结果不同（差约2.8%）。用户调节滑块时，如果总和≠100%走 estimateFromScratch、总和=100%走 trendData，会导致年化跳跃。

**当前方案**：统一使用 `estimateFromScratch`（各资产独立月度收益率加权），所有配置结果一致。三档方案的指标也基于此算法重新计算。

**数据更新**：
| 方案 | 年化 | 回撤 |
|---|---|---|
| 保守型 | 7.0% | -5.2% |
| 稳健型 | 10.4% | -10.5% |
| 进取型 | 19.6% | -21.0% |

**滑块交互**：各滑块独立调节，总和可 < 100%，缺额视为活期（0收益）。减现金=放活期，年化微降而非上升。

### 6.2 trendData 覆盖不完整

- 194条记录（191条粗扫描 + 3条精确方案）
- 大部分配置走 `estimateFromScratch`（真实数据加权），结果正确但和 trendData 精确回测结果可能有微小偏差
- 三档方案已注入确保精确匹配

### 6.3 Excel 下载链接

- 已修正为相对路径 `恒市值法理财操作表（稳健型）.xlsx`
- 文件在仓库根目录，GitHub Pages 部署后可直接下载
- 如需更新 Excel 内容，需重新上传并替换

### 6.4 主题切换时图表重建

- 主题切换会 dispose 所有 ECharts 实例并重建
- 如果添加新的图表容器，需要在 main.js `chartIds` 数组中注册
